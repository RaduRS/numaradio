
## Purpose
This document captures the current agreed direction for the AI radio project before moving into database schema design, queue rules, webhook maps, and implementation details. It is intended to act as the definitive planning brief for the first architecture phase.

The product direction described here is a global AI radio station with a real station identity, a continuous music stream, live listener interaction, background request generation, synthetic host segments, and a strong emphasis on personality without turning the product into a chaotic open chat system.

## Core concept
The project is a single public website that hosts a single global radio stream. Everyone hears the same station output at the same time. This is not a personalized playback product and it is not a private user-by-user music app.

The station continuously plays from a seeded music library. That library is expected to contain roughly 50 songs at launch, enough to loop continuously and maintain a proper radio feel without ever waiting on live generation. The catalog is expected to grow over time.

Listener requests are not meant to replace the main library. Instead, a listener can request a personalized song, and while the station continues to play the existing library, the request is processed in the background. As soon as the requested track is generated, stored, and marked ready, it can be inserted into the live station queue at the next appropriate point.

## Product identity
A single shared brand should be used for both the website and the radio station. The website and station should not have separate names because the website's main role is to host the stream, capture interaction, and reflect the station identity.

Shortlisted naming directions include:

- RequestFM
- NeonAir
- PromptFM
- EchoFM
- QueueFM
- AirMuse
- SongRush

The strongest recommendations so far are **RequestFM** for clarity and **NeonAir** for atmosphere and brand feel.

## Product positioning
The station should feel like an actual radio brand, not just a front end for AI music generation. That means the value is not only in generating songs on demand but in combining:

- A continuous shared stream.
- A recognizable host voice and station personality.
- Listener requests and shoutouts.
- AI-generated custom songs that can enter live rotation.
- Dynamic artwork and metadata for each track.

The host personality is important, but music remains central. The desired balance is personality-first in presentation, while still keeping the station fundamentally music-driven.

## Broadcast model
The station is one continuous global broadcast lane.

Key decisions:

- The stream is the same for everyone.
- The station does not pause or stall for generation.
- The library provides the baseline continuity.
- User-generated songs enter the live queue when they are actually ready.
- There is no per-user playback session.
- There is no mid-song AI voice speaking over tracks.
- The host speaks before and after songs only.

This is intentionally closer to a real radio station than to a playlist product or a one-to-one conversational assistant.

## Music sources
There are two primary music sources in the current plan.

### 1. Seed library
The seed library is made up of approved, already-available tracks stored in Backblaze B2 with track metadata stored in Neon. These tracks keep the station alive continuously.

This seed library can include tracks manually created in Suno and manually uploaded afterward. Suno is being treated as a manual import source, not as the live programmatic generation engine, because current public discussion still points developers toward unofficial or workaround-style API paths rather than a widely established official public integration route.[1][2]

### 2. Listener-requested generated tracks
When a user submits a request, the requested song is generated asynchronously in the background using MiniMax 2.6 as the working generation engine. The resulting audio is uploaded to B2, the metadata is stored in Neon, and the track is then made available for insertion into the live queue.

The generation pipeline does not replace the seed library. It adds freshness and personalization to a station that must already sound alive even before any user requests arrive.

## Synthetic host strategy
The station will use Deepgram for host-style voice inserts and short DJ segments.

Deepgram publicly positions separate products for Voice Agent behavior and for text-to-speech, and the pricing references available publicly distinguish the Voice Agent API from plain TTS/Aura-style speech generation.[3][4][5] Deepgram also documents real-time TTS and WebSocket-based speech streaming, which means spoken output can be generated quickly without having to batch long files first.[6][7][8]

The project direction is to use Deepgram mainly as a station host voice system rather than as an open-ended conversational agent running the entire product. In practice, that means the preferred station behavior is:

- Short intros before songs.
- Short outros after songs.
- Station IDs.
- Shoutouts.
- Weather/time/energy updates.
- Brief queue-related spoken moments.

The station should not sound like a phone agent. It should sound like a radio presenter. The behavior should be structured, event-based, and fast.

## Whether to store Deepgram inserts
A key design question was whether Deepgram speech must always be stored as files or whether it can be spoken live directly to the site.

The answer is mixed.

Deepgram supports streaming TTS behavior, so technically it is possible to generate and play speech live rather than permanently store every spoken segment.[6][7][9] However, for a radio station, timing and reliability matter more than theoretical flexibility.

The agreed direction is:

- Do **not** store every host utterance forever.
- Do use short-lived caching or file generation for queue-critical inserts that become part of the broadcast lane.
- Do allow more ephemeral speech handling for non-critical experiences if needed later.

In other words, station-critical intros and linked audio segments should usually be rendered into a broadcast-safe asset before playback. They can expire later, but they should not rely on fragile browser timing.

## Personality design
The station host is important, but it must be controlled.

The desired personality style is:

- Human radio style.
- Energetic but not intrusive.
- Short and timely.
- Reactive to what just played or what is coming next.
- Capable of giving the station a living identity.

The following are desired:

- Pre-song intros.
- Post-song reactions.
- Station IDs.
- Lightweight weather and status updates.
- Occasional hype or transition phrases.

The following are not desired:

- Mid-song voice interruptions.
- Endless monologues.
- Constant chatter.
- Raw user text spoken verbatim.
- A fully autonomous improvising host at launch.

The host should behave more like a well-produced DJ system than an uncontrolled AI personality.

## Website experience
The website is the station interface.

Core public website features should include:

- Live station player.
- Current track metadata.
- Current track artwork.
- Upvote and downvote reactions for the currently playing song.
- A request form for personalized songs.
- A shoutout form.
- A visible status for request progress.
- A recently played or recently requested section.

The website should feel simple, fast, and station-first. It should not feel like a complicated SaaS dashboard. The player and current experience should dominate the page.

## User accounts
The current planning direction does not require user registration for interaction.

This means the site should allow low-friction engagement from anonymous users, including:

- Reacting to the current song.
- Submitting a song request.
- Sending a shoutout.

Because there is no user registration, the system must rely more heavily on automatic safety and abuse controls.

## Requests and shoutouts
Requests and shoutouts should feel casual and easy, but they cannot be fully unfiltered.

The goal is not to make the forms feel strict or hostile. The goal is to make them feel radio-like while silently applying safety controls behind the scenes.

The agreed moderation style is:

- No human approval step.
- Strong automatic policy enforcement.
- Sanitization before generation or speech.
- Pornography blocked.
- Slurs and abusive content blocked.
- Doxxing or personal data blocked.
- Impersonation and risky abuse patterns blocked.
- Spam, flooding, and duplicate abuse suppressed.

The station should avoid a full anonymous public comment wall at launch. A free-form always-visible public comment feed introduces too much moderation pain for too little value. Requests and shoutouts are higher-value and much easier to control.

## Safe handling of user text
Raw user text should not flow directly into speech or generation without processing.

Instead, the system should:

1. Accept user input.
2. Validate length and basic structure.
3. Apply automatic filtering and safety scoring.
4. Rewrite or template the text into a safe broadcast form.
5. Pass only the cleaned version into the host script or generation prompt.

Example of the intended approach:

- User input: a messy or overlong shoutout.
- Broadcast-safe output: a short clean station phrase in the voice of the host.

This keeps the public interface open while preventing the station from broadcasting unsafe raw text.

## Continuous library-first playback
One of the most important clarified decisions is that the stream never waits for a requested song to be generated.

The 50-song library is the always-on base. The request pipeline runs in the background. While current songs continue playing from the queue, requested songs are generated in parallel. Once ready, they are inserted into an appropriate upcoming slot.

This makes the station resilient. It also matches the original intended concept more closely than any queue design that blocks playback on generation.

## Request pipeline
The request pipeline is central to the product.

High-level request lifecycle:

1. A listener submits a request.
2. The request is accepted by the website and written into Neon.
3. The request enters a pending or queued state.
4. NanoClaw picks up the request.
5. NanoClaw validates and sanitizes the input.
6. NanoClaw transforms the request into a generation-ready prompt package.
7. MiniMax 2.6 generates the song.
8. Artwork generation runs for the track.
9. Assets are uploaded to Backblaze B2.
10. Metadata and status are written back to Neon.
11. The track is marked ready.
12. The broadcast queue inserts the ready track at the next sensible point.
13. Deepgram produces the introduction or transition speech.
14. The song airs globally.

The system must support multiple simultaneous requests. The request queue is therefore not optional. If ten users request songs at once, they must enter a real background pipeline rather than all competing for immediate synchronous execution.

## Generation engine direction
MiniMax 2.6 is the working music generation engine for the on-demand request path. This direction aligns with prior experimentation around MiniMax music formatting and the broader preference to rely on a controllable generation workflow rather than depend on Suno as a live API backbone.

Suno remains useful as a manual content creation source for seeding the catalog. But the live architecture should not assume Suno behaves like a normal stable public developer API, because public materials still frame Suno integration in unofficial or workaround-driven terms rather than a broadly established first-party developer route.[1][2]

## Artwork generation
Each song should have custom artwork generated and associated with it.

Artwork matters because:

- It gives each track identity on the site.
- It makes the station feel richer and more alive.
- It increases the feeling that a request turned into a real broadcast artifact.
- It creates reusable assets for promotion and social clips later.

Artwork should be treated as a first-class asset, stored in B2 and linked through Neon metadata.

## Data and storage direction
The high-level storage plan is already clear.

### Neon
Neon will be the main operational database. It should store the relational state of the product, including:

- Tracks.
- Track metadata.
- Queue state.
- Request state.
- Shoutouts.
- Votes.
- Playback history.
- Asset references.
- Moderation and status fields.

### Backblaze B2
Backblaze B2 will be the main asset store for:

- Audio files.
- Artwork.
- Short-lived or cached host inserts where needed.
- Other derived media assets.

The website should not treat B2 as the main logic layer. B2 stores files. Neon stores state.

## NanoClaw's role
NanoClaw should be the orchestration layer, not the public website.

NanoClaw is the right place for:

- Picking up newly submitted requests.
- Running validation and moderation.
- Preparing generation prompts.
- Calling MiniMax 2.6 for song creation.
- Calling the artwork generation pipeline.
- Calling Deepgram for station segments.
- Uploading files to B2.
- Writing results to Neon.
- Updating request states.
- Pushing ready tracks into the broadcast queue.
- Handling retries and failure logic.

This is the correct use of NanoClaw because the core job is asynchronous orchestration across several external systems.

## What webhooks should do
Webhooks are useful, but they should be used carefully rather than everywhere.

The current direction is to use webhooks or equivalent event triggers for important state transitions such as:

- New request submitted.
- Generation completed.
- Artwork completed.
- Track marked ready.
- Track started playing.
- Track finished playing.
- Broadcast state updated.

Webhooks should not become the entire architecture. Many internal transitions can remain inside normal queue workers and application logic. The goal is event-driven clarity, not event-driven chaos.

## Global stream decision
A major strategic decision is that the station is global, not individualized.

This means:

- Everyone hears the same track order.
- A user request, once aired, becomes part of the shared station experience.
- The station behaves like radio, not like personal on-demand playback.
- The queue is optimized for station flow, not for individual response promises.

This is important because it affects every later decision: queue design, player logic, metadata updates, and expectation-setting on the public site.

## No human approval step
There should be no human approval step before airing request-driven songs or shoutouts.

However, this does **not** mean the station is unmoderated. It means moderation is automatic and policy-driven. The correct framing is:

- No human moderation in the normal path.
- Strong machine moderation and sanitization before generation and broadcast.

This preserves speed while still protecting the station.

## Station pacing philosophy
The station should feel alive and responsive, but not overloaded.

The preferred pacing style is:

- Song.
- Short host segment.
- Song.
- Short host segment.
- Occasional shoutout or themed transition.

The host should not be present between every single moment in a way that becomes repetitive. The host should enhance the station's identity, not dominate it.

## Why no open public comment wall at launch
A visible open anonymous comment feed seems attractive at first, but it is not a good launch priority.

Reasons:

- It is far harder to moderate than structured requests and shoutouts.
- It creates spam and abuse vectors quickly.
- It adds little to the actual radio feel.
- It complicates the public interface.
- It is a distraction from the main value loop.

A better launch structure is:

- Request form.
- Shoutout form.
- Voting on current track.
- Recent requests or recent plays.

This keeps the station interactive without opening a low-value moderation hole.

## Rights and reuse position
The current operating assumption is permissive downstream use from the station side. The product owner does not intend to lock listener reuse down.

At the same time, the architecture and public communication should still avoid overstating legal certainty around AI music ownership. Earlier research indicated that Suno distinguishes commercial-use rights on paid-plan outputs from broader certainty about copyright protection of fully AI-generated works.[10][11][12]

For architecture purposes, this means:

- Track provenance should still be retained.
- Source generator and creation context should remain traceable.
- Public messaging should be permission-based rather than built on strong claims of universal copyright exclusivity.

This document is not a legal brief, but the rights posture still affects system design and metadata.

## Broadcast-safe philosophy
A recurring theme across all decisions is the need for broadcast-safe behavior.

That means:

- The player should always have content to play.
- The queue should not depend on fragile synchronous generation.
- User text should always be cleaned before public use.
- Host speech should be controlled and concise.
- Station-critical audio should be reproducible and reliable.
- The whole product should feel stable, even if requests or generation jobs fail.

## What should not be built in the first release
The following are intentionally not part of the current first-phase direction:

- Per-user custom streams.
- Mid-song talking by the AI host.
- Endless autonomous AI monologues.
- Heavy public social commenting.
- A requirement for account creation.
- Human moderation as a normal workflow.
- Dependence on Suno as the live API generation engine.

These decisions help keep the architecture focused and launchable.

## Final agreed summary
The project is a single-brand, single-site, global AI radio station built around a continuous seeded song library, listener requests, async song generation, AI host speech, dynamic artwork, and automatic safety enforcement.

The station will:

- Stream globally to everyone.
- Use a seeded ~50-song library at launch to ensure uninterrupted playback, growing over time.
- Accept anonymous requests, votes, and shoutouts.
- Process requests through a real queue and pipeline.
- Generate requested tracks asynchronously using MiniMax 2.6.
- Use Suno as a manual library-creation source rather than the live API backbone.
- Use Deepgram primarily for structured host speech and transition segments rather than as a free-form always-on agent.
- Store files in Backblaze B2 and state in Neon.
- Use NanoClaw for orchestration, retries, and workflow control.
- Apply automatic policy enforcement instead of human moderation.
- Keep the host out of the middle of songs.
- Sound like a real station rather than a demo of disconnected AI tools.
