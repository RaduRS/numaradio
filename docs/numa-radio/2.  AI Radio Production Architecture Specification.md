
## Purpose
This document complements the earlier architecture brief and moves the project from concept definition into a production-oriented systems specification. The earlier document defined the product, experience, and operating principles. This document defines the exact production architecture shape, system components, data model, workflow boundaries, queue states, event flow, webhook map, and NanoClaw orchestration required to implement the station in a launch-ready way.

This document is still intentionally one level above source code. It does not provide SQL migrations, worker code, or deployment manifests. Instead, it defines the exact implementation blueprint those later artifacts should follow.

## Relationship to the first document
The first architecture brief established the following high-level decisions:

- One brand, one site, one global station.
- Continuous library-first playback.
- MiniMax 2.6 for request-based generation.
- Suno as manual catalog input rather than the live API backbone.[1][2]
- Deepgram as the host speech layer, used mainly for structured radio presentation rather than an uncontrolled always-on agent.[3][4]
- Neon as the operational database.
- Backblaze B2 as the file store.
- NanoClaw as the orchestration engine.
- Automatic policy enforcement instead of human moderation.
- No mid-song voice interruptions.

This second document assumes those decisions are locked and expands them into the exact production architecture.

## System overview
At production level, the system should be treated as six major layers:

1. Public web application.
2. Core application API.
3. Operational database and cache/state layer.
4. Asset storage layer.
5. Workflow orchestration layer.
6. Broadcast assembly and stream delivery layer.

### Public web application
The website is the listener-facing control surface. Its responsibilities are:

- Render the player.
- Display the current track and artwork.
- Show recent and upcoming activity.
- Accept anonymous requests.
- Accept shoutouts.
- Accept thumbs up and thumbs down input.
- Show request status after submission.
- Display the global station identity.

The website should not contain the orchestration logic. It only interacts with the API and reflects station state.

### Core application API
The API is the system of record interface between the web app and the backend. It should:

- Accept public requests and shoutouts.
- Validate payload shape.
- Write submission rows to Neon.
- Create initial request events.
- Trigger downstream workflow events.
- Read current station state for the front end.
- Expose the current track, recent plays, and request status.
- Accept vote submissions.
- Provide internal endpoints for workflow callbacks.

The API should be stateless and should not attempt to perform generation inline.

### Operational database
Neon is the operational state layer. Neon should hold the state of the station, not the actual audio payloads. Neon is responsible for:

- Canonical track metadata.
- Request lifecycle state.
- Queue planning state.
- Playback state.
- Asset references.
- User interaction state.
- Moderation results.
- Workflow history.

### Asset storage
Backblaze B2 is the durable asset store. It should contain:

- Master audio files.
- Derived stream-friendly audio files if needed.
- Artwork images.
- Temporary Deepgram host inserts that become part of the broadcast chain.
- Optional waveform or promo derivatives.
- Metadata snapshots where useful.

### Workflow orchestration
NanoClaw is the async orchestration engine. It is responsible for:

- Pulling pending requests.
- Applying safety and prompt transformation.
- Calling generation providers.
- Handling retries and fallback paths.
- Creating track records.
- Uploading outputs.
- Updating state in Neon.
- Triggering queue insertion.
- Triggering host segment generation.

### Broadcast layer
The broadcast layer is responsible for actually producing the station output. It should:

- Determine what airs next.
- Join host inserts and songs into the current play order.
- Maintain a forward queue buffer.
- Update current playback state.
- Emit playback events for the website and internal systems.
- Deliver a stable global stream.

**Implementation (locked 2026-04-19): Icecast + Liquidsoap** running on the home mini-server. Liquidsoap holds the playout queue, crossfades between segments, and splices in pre-rendered Deepgram host inserts. Icecast serves the resulting MP3 stream. Listeners reach it at `api.numaradio.com/stream` via Cloudflare Tunnel. The broadcast worker described later in this document is therefore implemented largely as a Liquidsoap script plus a thin TypeScript shim that emits playback events back to the API as segments transition.

## Recommended deployment shape
A practical production deployment should be split into these deployable services:

| Service | Main role |
|---|---|
| `web` | Public website and player UI (Vercel) |
| `api` | Public and internal application API (Vercel) |
| `worker-orchestrator` | NanoClaw workflow runner (mini-server) |
| `worker-queue` | Queue planner and queue maintenance (mini-server) |
| `worker-broadcast` | Liquidsoap + Icecast — playout, crossfade, host-insert splicing, MP3 stream serving (mini-server) |
| `worker-cleanup` | Expiry, retries, and cleanup tasks (mini-server) |
| `db` | Neon |
| `storage` | Backblaze B2 |

This keeps the architecture clean and makes it easier to isolate failures. A request-generation failure should not crash the public player. A broadcast event issue should not stop request intake.

## Core domain model
The production model revolves around six core domain objects:

1. Track.
2. Request.
3. Shoutout.
4. Queue item.
5. Broadcast segment.
6. Playback event.

### Track
A track is a playable song in the station library. A track may come from:

- Manual Suno import.
- MiniMax-generated request output.
- Future external catalog ingestion.

A track is the canonical music object, even if it has multiple associated files or variants.

### Request
A request is a listener's submission asking the station to create or air something. A request becomes either:

- A generated song.
- A rejected submission.
- A duplicate of something already in progress.
- A sanitized and accepted pipeline job.

### Shoutout
A shoutout is structured listener text intended to become a station-safe host mention. It is distinct from a request because it may not require song generation.

### Queue item
A queue item is a scheduling object representing something planned for future playback. The queue may contain:

- A track.
- A host intro.
- A host outro.
- A station ID.
- A shoutout segment.
- A weather or status insert.

### Broadcast segment
A broadcast segment is the exact unit the broadcast layer can play. It is more literal than a queue item. For example, a single queue item representing “play a requested track with intro” may resolve into two broadcast segments:

- Host intro audio.
- Track audio.

### Playback event
A playback event is an immutable log row recording something that happened on air, such as:

- Track started.
- Track ended.
- Host insert started.
- Host insert ended.
- Queue advanced.

## Relationship summary
A practical relationship map looks like this:

- One station has many tracks.
- One station has many requests.
- One station has many shoutouts.
- One request may produce one track.
- One track has many assets.
- One track may create several queue items over time.
- One queue item resolves into one or more broadcast segments.
- Broadcast segments write to play history.
- Requests and shoutouts create moderation flags and workflow runs.

- `ready_for_queue`: track is production-ready.

## Queue strategy
The station queue should be divided conceptually into three layers:

1. Library rotation layer.
2. Request injection layer.
3. Host and utility insert layer.

### Library rotation layer
This is the baseline music rotation. It ensures the station always has enough songs ready to play.

Responsibilities:

- Maintain a healthy forward queue buffer.
- Avoid repeating the same tracks too closely.
- Support mood and genre diversity later.
- Guarantee continuity if request generation is slow.

### Request injection layer
This layer introduces newly generated request songs. It should:

- Prioritize fresh generated songs enough that users feel momentum.
- Avoid overwhelming the station with back-to-back request songs.
- Blend request content into the station rhythm.
- Allow urgent high-priority inserts if desired later.

### Host and utility insert layer
This layer adds:

- Intros.
- Outros.
- Station IDs.
- Shoutouts.
- Weather updates.
- Mood or energy transitions.

These should be treated as queue-managed playout objects, not random front-end effects.

## Recommended queue rules
This is the starting production logic for queue planning.

### Queue buffer rules
- Maintain at least 30 minutes of fully staged upcoming content.
- Aim for 45 to 60 minutes of planned queue depth.
- Trigger queue refill whenever staged depth drops below threshold.

### Request insertion rules
- Newly ready request songs should enter the near-term queue, not be dumped into a distant backlog.
- Do not place two request songs back-to-back by default.
- Do not place a request song directly after a song by the same mood/genre tag if a better transition exists.
- Favor insertion after one standard library track or one host segment.
- Allow a request song to air in the next 1 to 3 suitable slots if the queue depth permits.

### Host insert rules
- Prefer one short intro before a featured request song.
- Use outros selectively, not after every single track.
- Insert a station ID every few songs or at time-based intervals.
- Limit total spoken interruptions so the station still feels music-forward.

### Safety fallback rules
- If a request track is ready but its intro asset failed, allow the track to air without the intro.
- If shoutout generation fails, drop the shoutout and preserve music continuity.
- If artwork generation fails, the track may still air using a fallback artwork asset.

## Broadcast timing rules
Because this is a true global stream, timing matters.

### Broadcast rules
- Only one segment can be active per station at any moment.
- Every segment must have a clear start and end event.
- `now_playing` must update at segment boundaries.
- The broadcast worker should never wait for a just-in-time generation call if a prepared asset exists.
- Host speech for the upcoming segment should be staged before the currently playing music ends whenever possible.

### Segment assembly rules
Before the next block airs, the system should know:

- Which segment is next.
- Which audio asset it will use.
- Exact estimated duration.
- Whether a host intro exists.
- Whether the website can update visuals in sync.

### Event: `request.sanitized`
Request text has passed validation and prompt shaping.

### Event: `request.rejected`
Request blocked by safety or abuse checks.

### Event: `request.song_generation_started`
Song generation in progress.

### Event: `request.song_generation_completed`
Song audio generated successfully.

### Event: `request.song_generation_failed`

## Queue-originated events
These come from the queue planner.

### Event: `queue.item_inserted`
A queue item was inserted into the station queue.

### Event: `queue.item_staged`
A queue item was resolved to a playout-ready asset chain.

### Event: `queue.item_failed`
Queue staging failed.

### Event: `queue.refill_needed`
Forward queue depth dropped below threshold.

## Broadcast-originated events
These come from the broadcast worker (Liquidsoap + thin shim).

### Event: `broadcast.segment_started`
A specific segment began.

### Event: `broadcast.segment_completed`
A specific segment ended normally.

### Event: `broadcast.segment_failed`
A segment failed during playback or could not be loaded.

### Event: `broadcast.track_started`
A music track began.

### Event: `broadcast.track_completed`
A music track ended.

### Event: `broadcast.now_playing_updated`
Current station state changed and the website should refresh public now-playing data.

## Where true webhooks should be used
Architecturally, not all events need external HTTP webhooks. Some can stay in the database or internal queue. The real outbound/inbound webhook boundaries should be:

- App API -> orchestration service when a request or shoutout is submitted.
- External generation service -> app callback if the provider uses async completion callbacks.
- Broadcast service -> app callback when playback state changes.
- Optional analytics/social automations -> internal event consumers.

If a provider does not support callbacks, NanoClaw should poll instead of forcing artificial webhook complexity.

## NanoClaw workflow design
NanoClaw is the heart of the async system. It should be organized as named workflows rather than as one giant worker.

Recommended workflows:

- Request generation workflow.
- Shoutout preparation workflow.
- Track ingest workflow.
- Queue repair workflow.
- Host insert generation workflow.
- Asset cleanup workflow.

## Workflow 1: request generation
This is the main workflow for personalized song creation.

### Step 1: claim request
- Select request rows in `queued` state.
- Lock one request for processing.
- Create a `workflow_runs` row.
- Move request to `sanitizing`.

### Step 2: policy enforcement
- Validate length, rate limits, duplicate risk, blocked terms, sexual content, harassment, personal data, and other unsafe patterns.
- Write `moderation_flags` as needed.
- If blocked, move request to `rejected_policy` and emit `request.rejected`.
- If rate-limited, move request to `rate_limited`.

### Step 3: sanitize and structure
- Normalize the request into a clean generation brief.
- Extract optional fields such as name, mood, genre, dedication target, or occasion.
- Write cleaned result back to `clean_text` and `prompt_payload_json`.
- Move request to `prompt_ready`.
- Emit `request.sanitized`.

### Step 4: deduplication
- Compute a dedupe key.
- Check for very recent identical or near-identical requests already in progress.
- If duplicate, either attach the request to an existing pending result or continue based on policy.
- If attached to an existing request, mark `deduplicated` and stop further generation.

### Step 5: generate song
- Move request to `generating_song`.
- Call MiniMax 2.6 generation workflow with the prompt package.
- Record attempt count.
- If transient failure occurs, retry according to policy.
- If permanent failure occurs, set `song_failed` and finish workflow with failure.
- On success, stage the returned audio asset for upload.
- Emit `request.song_generation_completed`.

### Step 6: generate artwork
- Move request to `generating_art`.
- Generate artwork from the request theme, cleaned prompt, or resulting song metadata.
- On repeated failure, either set `art_failed` with fallback art allowed or abort according to policy.
- Emit `request.art_generation_completed` on success.

### Step 7: assemble track
- Move request to `assembling_track`.
- Upload audio and artwork to B2.
- Create `tracks` row.
- Create `track_assets` rows.
- Link `result_track_id` back to `requests`.
- Set `tracks.track_status = ready` and `requests.request_status = ready_for_queue`.
- Emit `request.track_assembled` and `request.ready_for_queue`.

### Step 8: trigger queue insertion
- Hand off to queue planner.
- Queue planner creates one or more `queue_items`.
- Request moves to `queued_for_air`.

### Step 9: close workflow
- Mark workflow completed.
- Persist final result payload.

## Workflow 2: shoutout preparation
This workflow is simpler and faster than full request generation.

### Step 1: claim shoutout
- Select pending shoutout.
- Lock it.
- Create workflow run.

### Step 2: policy enforcement
- Run content filter, anti-abuse rules, and length limits.
- If unsafe, mark `rejected_policy`.

### Step 3: rewrite for air
- Transform raw user text into a short, broadcast-safe phrase.
- Limit duration target aggressively.
- Save `broadcast_text`.

### Step 4: optional TTS pre-generation
- If the shoutout is selected for actual airplay, generate Deepgram TTS and store the asset.[4][5][6]
- If not selected immediately, keep text ready and allow later TTS generation.

### Step 5: queue handoff
- Mark shoutout `ready`.
- Insert queue item when rules permit.

## Workflow 3: track ingest
This workflow handles manual catalog additions such as Suno imports.

### Step 1: intake metadata
- Receive uploaded file reference and metadata.
- Create ingest workflow run.

### Step 2: upload or verify B2 asset
- Move file into canonical B2 path if needed.
- Record asset details.

### Step 3: normalize track metadata
- Standardize title, duration, genre tags, and provenance.
- Mark `source_type = suno_manual` where applicable.[1][2]

### Step 4: approve for library
- Set `airing_policy = library`.
- Set track status `ready`.

This workflow is important because the 50-song seed library depends on reliable manual ingestion.

## Workflow 4: host insert generation
This workflow creates track-linked or station-linked host segments.

### Step 1: determine need
Generate host speech only when needed, such as:

- before featured request tracks
- after major transitions
- station ID intervals
- weather/time inserts
- selected shoutouts

### Step 2: compose script
- Use structured templates.
- Pull only sanitized inputs.
- Never pass raw user text directly.
- Keep word count short.

### Step 3: call Deepgram TTS
- Generate the audio segment.
- Store a short-lived asset if this insert is entering the queue.[4][5][6]

### Step 4: create queue item or segment
- Either create a standalone queue item or attach the asset to an existing track play plan.

## Workflow 5: queue repair
This workflow protects uptime.

When triggered:

- queue depth too low
- staging failed
- asset missing
- current queue contains invalid object

Actions:

- remove dead queue items
- replace with library tracks
- regenerate host insert if optional
- restore forward buffer quickly

This workflow ensures the station never goes silent just because one request pipeline failed.

## Queue planner step-by-step
The queue planner deserves its own explicit logic.

### Step 1: inspect forward buffer
- Calculate currently staged minutes.
- Calculate planned but unstaged minutes.
- If staged minutes below threshold, continue refill.

### Step 2: inspect urgent ready requests
- Find requests in `ready_for_queue` not yet inserted.
- Rank by freshness, fairness, and current station fit.

### Step 3: inspect standard library needs
- Pull suitable library tracks to maintain continuity.
- Avoid near-term repeats.

### Step 4: determine insert pattern
A recommended default pattern is:

- library track
- optional host insert
- request track
- library track
- optional station ID
- library track

This pattern should remain flexible but prevents request songs from flooding the station.

### Step 5: create queue items
- Insert queue rows with `planned` status.
- Assign `position_index` values.
- Create linkage between request-driven intros and target tracks when needed.

### Step 6: stage near-term items
- Resolve the next items to actual assets.
- Create `broadcast_segments` rows pointing at downloadable B2 URLs (Liquidsoap will fetch and play these).
- Mark queue items `staged`.

## Broadcast worker step-by-step
The broadcast worker is the system that actually turns planned content into a real stream.

**Implementation note:** in this architecture the "broadcast worker" is **Liquidsoap** running on the home mini-server, plus a small TypeScript shim that publishes playback events back to the API. Liquidsoap is configured to read upcoming `staged` segments (via a small request handler the shim exposes), play them sequentially through a crossfade chain, and call back when each segment starts and ends. Icecast sits in front of Liquidsoap and serves the resulting MP3 stream to listeners.

The conceptual steps below still apply — they're just executed by the Liquidsoap script + shim instead of bespoke worker code.

### Step 1: pick next staged segment
- Select next `broadcast_segments` row in order.
- Confirm asset exists.
- Confirm not already started.

### Step 2: update current pointers
- Update `now_playing`.
- Emit `broadcast.segment_started`.
- If segment is music, emit `broadcast.track_started`.

### Step 3: play asset
- Push audio into the station stream.
- Maintain heartbeat while playing.

### Step 4: on completion
- Mark segment complete.
- Write to `play_history`.
- Update queue item state.
- Emit `broadcast.segment_completed`.
- If music, emit `broadcast.track_completed`.

### Step 5: advance queue
- Trigger queue refill if needed.
- Update public now-playing event.

- Easier cleanup.
- Easier debugging.

## Anti-abuse and safety controls
Because there are no user accounts, abuse resistance must be designed into the system.

Recommended controls:

- IP hash rate limiting.
- Browser fingerprint rate limiting.
- Request text length caps.
- Duplicate content suppression.
- Cooldown windows per fingerprint.
- Profanity and unsafe-content blocking.
- Personal data detection.
- Impersonation and protected-term screening.
- Request budget per time window.
- Global circuit breaker during abuse spikes.

A request should be blocked early whenever possible, before expensive generation begins.

## Failure handling
A launch-ready station must define failure behavior explicitly.

### Song generation failure
- Retry transient failures.
- If still failed, mark request `song_failed`.
- Do not allow the queue to wait on the request.
- Preserve request status for the user.

### Artwork failure
- Retry once or twice.
- If still failed, use fallback artwork.
- Do not block the song from airing unless policy requires artwork.

### Deepgram host failure
- Drop optional host insert.
- Continue with music.
- Log failure for later diagnostics.[4][6]

### Queue corruption or underflow
- Trigger `queue_repair`.
- Pull immediate fallback library tracks.
- Restore minimum staged buffer.

### Missing asset in B2
- Mark queue item failed.
- Replace with fallback item.
- Create workflow event for investigation.

### Liquidsoap or Icecast process death
- Process supervisor (systemd unit) auto-restarts.
- On restart, Liquidsoap rejoins from the latest `staged` segment.
- Brief silence (~5–15s) is acceptable; fallback rotation file plays if no staged segment is available.

To build efficiently, implementation should happen in this order:

This order makes sure the station can already play reliably before the more advanced request logic is turned on.

## Explicit non-goals for this phase
This production architecture still intentionally excludes:

- per-user streams
- live mid-song AI speech
- long-form autonomous DJ conversation
- public anonymous comment threads
- user accounts as a required dependency
- manual approval as the normal path
- dependence on Suno for live programmatic generation
- listener-pasted Suno URLs (dropped 2026-04-19, see Decisions Log)

These exclusions remain important because they keep the system launchable within the current scope.[1][2]

## Final implementation picture
When all pieces are working together, the system should operate like this:

1. The station has a seeded catalog of ready library tracks in Neon and B2.
2. The queue planner keeps a forward queue of songs and inserts.
3. Liquidsoap continuously plays staged segments and Icecast serves the stream.
4. The website reflects now-playing state and accepts anonymous interaction.
5. A listener submits a request.
6. The API stores the request and emits `request.submitted`.
7. NanoClaw claims the request, sanitizes it, moderates it, and builds a generation package.
8. MiniMax 2.6 generates the requested song.
9. Artwork is generated.
10. Assets are uploaded to B2 and linked in Neon.
11. The request track is marked ready and inserted into the live queue.
12. A short Deepgram intro is optionally generated and staged.
13. The request song airs as part of the same global station everyone is hearing.
14. Play history, request status, and current station state update automatically.
