## Purpose
This document is the final implementation blueprint that follows the earlier architecture brief and production architecture specification. The first document defined the product and operating model. The second document defined the production system shape. This document turns those decisions into implementation-ready structures: data models, event contracts, queue logic, workflow pseudocode, API contracts, storage key patterns, retry policy, deployment topology, and environment configuration.

This document should be treated as the working reference for building the first launch-ready version of the station.

## Scope lock
The following assumptions are treated as fixed for this implementation blueprint:

- One global station.
- One shared website and brand.
- Library-first continuous playback.
- Manual Suno ingest for seeded catalog content, not live programmatic Suno generation.[1][2]
- MiniMax 2.6 as the request-song generation engine.
- Deepgram for structured host speech and station inserts.[3][4][5]
- Neon as the operational database.
- Backblaze B2 as asset storage.
- NanoClaw as the orchestration engine.
- No human approval in the normal path.
- Automatic policy enforcement.
- No mid-song AI voice.

## Final stack decision
The implementation stack should be:

| Layer | Choice |
|---|---|
| Front end | Next.js |
| Public/API hosting | Vercel |
| Database | Neon Postgres |
| File storage | Backblaze B2 |
| Orchestration | NanoClaw worker service |
| Song generation | MiniMax 2.6 |
| Host voice | Deepgram TTS / host inserts |
| Image generation | Separate image generation provider |
| Broadcast engine | Always-on worker/container service |
| Observability | Logs + metrics + workflow/event tables |

## Vercel deployment decision
Vercel is appropriate for the website, API routes, status endpoints, and admin UI. Vercel should not be treated as the only runtime for the entire system because long-running orchestration, queue maintenance, and broadcast assembly are better handled by dedicated always-on workers.

The recommended deployment split is:

- Vercel for `web` and `api`.
- A separate always-on container host for `worker-orchestrator`, `worker-queue`, `worker-broadcast`, and `worker-cleanup`.
- Neon for relational state.
- Backblaze B2 for media files.

This keeps the public site fast while preserving stable long-running background execution.

## Services
The implementation should be split into these concrete services.

### 1. `web`
Responsibilities:
- Render station UI.
- Play global stream.
- Show now playing.
- Show recent plays and request status.
- Submit requests, shoutouts, and votes.

### 2. `api`
Responsibilities:
- Public request intake.
- Public shoutout intake.
- Vote intake.
- Now playing and recent-play endpoints.
- Request status endpoint.
- Internal event endpoints.

### 3. `worker-orchestrator`
Responsibilities:
- Claim request and shoutout jobs.
- Run moderation and sanitization.
- Call MiniMax and image generation.
- Call Deepgram when a host asset is needed.[3][4]
- Upload assets.
- Create tracks.
- Emit ready-for-queue events.

### 4. `worker-queue`
Responsibilities:
- Keep forward queue healthy.
- Insert ready request songs.
- Insert host segments and station IDs.
- Stage near-term playout objects.
- Recover from queue underflow.

### 5. `worker-broadcast`
Responsibilities:
- Resolve next broadcast segment.
- Update now-playing state.
- Write play history.
- Emit playback events.
- Maintain stream heartbeat.

### 6. `worker-cleanup`
Responsibilities:
- Expire short-lived Deepgram inserts.
- Retry stale workflows.
- Archive failed jobs.
- Repair broken queue references.
- Delete temporary intermediates.

## Request lifecycle contract
Final request states should be used exactly as follows:

| State | Meaning |
|---|---|
| `submitted` | API accepted the request |
| `rate_limited` | blocked by anti-abuse throttling |
| `rejected_policy` | blocked by safety rules |
| `queued` | accepted for background processing |
| `sanitizing` | being normalized and checked |
| `deduplicated` | attached to existing in-flight equivalent |
| `prompt_ready` | prompt package prepared |
| `generating_song` | MiniMax is generating audio |
| `song_failed` | generation failed after retry policy |
| `generating_art` | artwork being created |
| `art_failed` | art failed after retry policy |
| `assembling_track` | files uploading and records being linked |
| `ready_for_queue` | song is ready to be scheduled |
| `queued_for_air` | queue planner inserted it |
| `aired` | request result aired on station |
| `expired` | request aged out or was dropped |

## Queue insertion algorithm
This is the first concrete queue algorithm to implement.

### Inputs
- current staged queue depth
- planned queue depth
- ready request tracks
- current time
- play history window
- track rotation history
- shoutouts ready for air
- station utility inserts due

### Hard rules
1. Never allow staged depth to fall below 30 minutes.
2. Aim for 45 to 60 minutes of planned future content.
3. Never place two request tracks back-to-back by default.
4. Never place the same track again inside the repeat-protection window.
5. Never block the queue waiting for a request track.
6. If a request track is ready, try to place it within the next 1 to 3 suitable song slots.
7. If a host intro exists for a request track, it must be staged before the track itself.
8. If a host intro fails, the track may still air.
9. If a shoutout fails, drop it rather than delaying music.

### Repeat-protection defaults
```ts
const ROTATION_RULES = {
  sameTrackCooldownMinutes: 180,
  sameMoodSoftCooldownMinutes: 30,
  sameGenreSoftCooldownMinutes: 20,
  requestTrackSpacingSongs: 1,
  stationIdEveryNSongs: 4,
  targetStagedMinutesMin: 30,
  targetPlannedMinutesMax: 60,
};
```

### Queue planner pseudocode
```ts
async function refillQueue(stationId: string) {
  const staged = await getStagedMinutes(stationId);
  const planned = await getPlannedMinutes(stationId);

  if (planned >= ROTATION_RULES.targetPlannedMinutesMax) return;

  const readyRequests = await getReadyRequestTracks(stationId);
  const readyShoutouts = await getReadyShoutouts(stationId);
  const dueUtility = await getDueUtilityInserts(stationId);
  const recentHistory = await getRecentPlayHistory(stationId);

  let insertionCursor = await getNextQueuePosition(stationId);
  let songsSinceLastId = await getSongsSinceLastStationId(stationId);
  let lastWasRequest = await didLastPlannedSongComeFromRequest(stationId);

  while ((await getPlannedMinutes(stationId)) < ROTATION_RULES.targetPlannedMinutesMax) {
    if (songsSinceLastId >= ROTATION_RULES.stationIdEveryNSongs && dueUtility.stationId) {
      await insertUtilityItem(stationId, dueUtility.stationId, insertionCursor++);
      songsSinceLastId = 0;
      continue;
    }

    if (!lastWasRequest) {
      const requestTrack = pickBestRequestTrack(readyRequests, recentHistory);
      if (requestTrack) {
        const intro = await maybeCreateOrFetchHostIntro(requestTrack.id);
        if (intro) await insertHostIntro(stationId, requestTrack.id, intro.assetId, insertionCursor++);
        await insertTrackQueueItem(stationId, requestTrack.id, 'request_pipeline', insertionCursor++);
        await markRequestQueuedForAir(requestTrack.requestId);
        lastWasRequest = true;
        songsSinceLastId += 1;
        continue;
      }
    }

    const libraryTrack = await pickBestLibraryTrack(stationId, recentHistory);
    if (!libraryTrack) break;
    await insertTrackQueueItem(stationId, libraryTrack.id, 'scheduler', insertionCursor++);
    lastWasRequest = false;
    songsSinceLastId += 1;

    const shoutout = pickOptionalShoutout(readyShoutouts);
    if (shoutout && shouldPlaceShoutoutHere()) {
      const shoutoutAsset = await maybeCreateOrFetchShoutoutAsset(shoutout.id);
      if (shoutoutAsset) {
        await insertShoutoutQueueItem(stationId, shoutout.id, shoutoutAsset.id, insertionCursor++);
      }
    }
  }

  await stageNearTermQueueItems(stationId);
}
```

## Broadcast segment assembly rules
A queue item resolves to one or more broadcast segments.

### Music track with intro
- segment 1: host intro
- segment 2: track audio

### Plain library song
- segment 1: track audio

### Shoutout block
- segment 1: host shoutout audio

### Station ID block
- segment 1: station ID audio

### Segment assembly pseudocode
```ts
async function stageQueueItem(queueItemId: string) {
  const item = await getQueueItem(queueItemId);

  if (item.queueType === 'music' && item.trackId) {
    const introAsset = await getLinkedHostIntroAsset(item.trackId);
    let order = 1;

    if (introAsset) {
      await createBroadcastSegment({
        stationId: item.stationId,
        queueItemId: item.id,
        segmentType: 'audio_host',
        assetId: introAsset.id,
        segmentOrder: order++,
        durationSeconds: introAsset.durationSeconds ?? 8,
      });
    }

    const audioAsset = await getTrackStreamAsset(item.trackId);
    await createBroadcastSegment({
      stationId: item.stationId,
      queueItemId: item.id,
      segmentType: 'audio_track',
      assetId: audioAsset.id,
      segmentOrder: order,
      durationSeconds: audioAsset.durationSeconds ?? item.estimatedDurationSeconds ?? 180,
    });
  }

  if (item.queueType === 'shoutout' && item.assetId) {
    await createBroadcastSegment({
      stationId: item.stationId,
      queueItemId: item.id,
      segmentType: 'audio_host',
      assetId: item.assetId,
      segmentOrder: 1,
      durationSeconds: item.estimatedDurationSeconds ?? 6,
    });
  }

  await markQueueItemStaged(item.id);
}
```

## NanoClaw workflow pseudocode
The orchestrator should not be one giant function. Use named workflows.

## Workflow: request generation
```ts
async function runRequestGenerationWorkflow(requestId: string) {
  const run = await startWorkflowRun('request_generation', 'request', requestId);

  try {
    await transitionRequest(requestId, 'submitted', 'queued');
    await transitionRequest(requestId, 'queued', 'sanitizing');

    const request = await getRequest(requestId);
    const moderation = await moderateRequest(request.rawText, request.fingerprintHash, request.ipHash);

    if (moderation.decision === 'rate_limit') {
      await setRequestState(requestId, 'rate_limited', moderation.reason);
      await completeWorkflow(run.id, { outcome: 'rate_limited' });
      return;
    }

    if (moderation.decision === 'block') {
      await setRequestState(requestId, 'rejected_policy', moderation.reason);
      await createModerationFlags('request', requestId, moderation.flags);
      await completeWorkflow(run.id, { outcome: 'blocked' });
      return;
    }

    const clean = await sanitizeRequestText(request.rawText);
    const promptPayload = await buildPromptPayload(clean);
    await saveRequestSanitized(requestId, clean, promptPayload);
    await transitionRequest(requestId, 'sanitizing', 'prompt_ready');

    const duplicate = await findDuplicateActiveRequest(promptPayload.dedupeKey);
    if (duplicate) {
      await setRequestDeduplicated(requestId, duplicate.id);
      await completeWorkflow(run.id, { outcome: 'deduplicated', linkedTo: duplicate.id });
      return;
    }

    await transitionRequest(requestId, 'prompt_ready', 'generating_song');
    const song = await generateSongWithRetry(promptPayload);

    await transitionRequest(requestId, 'generating_song', 'generating_art');
    const art = await generateArtworkWithRetry(promptPayload, song);

    await transitionRequest(requestId, 'generating_art', 'assembling_track');
    const upload = await uploadGeneratedAssets({ song, art, requestId });
    const track = await createTrackFromRequest(requestId, upload, promptPayload);

    await transitionRequest(requestId, 'assembling_track', 'ready_for_queue');
    await emitSystemEvent('request.ready_for_queue', 'request', requestId, {
      requestId,
      trackId: track.id,
      stationId: track.stationId,
    });

    await completeWorkflow(run.id, { outcome: 'success', trackId: track.id });
  } catch (error: any) {
    await failWorkflow(run.id, error);
    await maybeMarkRequestFailed(requestId, error);
    throw error;
  }
}
```

## Workflow: shoutout preparation
```ts
async function runShoutoutWorkflow(shoutoutId: string) {
  const shoutout = await getShoutout(shoutoutId);
  const moderation = await moderateShoutout(shoutout.rawText, shoutout.fingerprintHash, shoutout.ipHash);

  if (moderation.decision !== 'allow') {
    await rejectShoutout(shoutoutId, moderation.reason);
    return;
  }

  const clean = await sanitizeShoutoutText(shoutout.rawText);
  const broadcastText = await rewriteForBroadcast(clean);
  await saveShoutoutPrepared(shoutoutId, clean, broadcastText);

  if (shouldQueueShoutoutNow()) {
    const asset = await generateDeepgramInsert(broadcastText, 'shoutout');
    const stored = await storeHostInsert(asset, shoutoutId);
    await queuePreparedShoutout(shoutoutId, stored.assetId);
  }
}
```

## Moderation pipeline contract

### Layer 1: intake limits
- required fields
- max text length

- IP throttle
- duplicate burst check

### Layer 2: blocklist and risky pattern scan
- slurs
- pornographic intent
- sexual content involving minors
- threats
- personal data
- impersonation phrases
- branded attack bait

### Layer 3: rewrite layer
- remove profanity if salvageable
- shorten to broadcast-safe phrasing
- template shoutouts into host-safe sentences
- remove raw user syntax that sounds awkward on air

### Layer 4: generator-safe prompt building
- strip unsafe or banned instruction patterns
- remove artist-cloning or direct impersonation framing if policy requires
- normalize style descriptors

### Failure policy summary
- song failure: keep request visible as failed, never block station
- art failure: use fallback artwork and continue
- Deepgram failure: skip optional insert and continue music
- queue staging failure: replace with library fallback item
- missing B2 asset: mark queue item failed and trigger queue repair

### Playback loop pseudocode
```ts
async function broadcastLoop(stationId: string) {
  while (true) {
    const segment = await getNextStagedSegment(stationId);

    if (!segment) {
      await triggerQueueRepair(stationId, 'missing_segment');
      await sleep(1000);
      continue;
    }

    try {
      await markSegmentStarted(segment.id);
      await updateNowPlayingFromSegment(segment.id);
      await emitSystemEvent('broadcast.segment_started', 'broadcast_segment', segment.id, {
        stationId,
        queueItemId: segment.queueItemId,
        segmentId: segment.id,
      });

      await playSegmentAsset(segment.assetId);

      await markSegmentCompleted(segment.id);
      await appendPlayHistory(segment.id);
      await emitSystemEvent('broadcast.segment_completed', 'broadcast_segment', segment.id, {
        stationId,
        queueItemId: segment.queueItemId,
        segmentId: segment.id,
      });
    } catch (error: any) {
      await markSegmentFailed(segment.id, error.message);
      await triggerQueueRepair(stationId, 'segment_failed');
    }
  }
}
```

function buildTrackIntro(input: {

## Seed library ingest rules
Manual Suno imports should follow a consistent ingest flow because Suno is being treated as a manual catalog source rather than the live programmatic engine.[1][2]

Required ingest metadata:
- original filename
- source platform
- created during paid subscription yes/no
- creation date if known
- genre
- mood
- duration
- title override if needed
- rights/provenance notes

Ingest checklist:
1. upload file to B2 canonical path
2. create `tracks` row with `sourceType = suno_manual`
3. create `track_assets` row for audio
4. attach artwork if available
5. set `trackStatus = ready`
6. set `airingPolicy = library`
